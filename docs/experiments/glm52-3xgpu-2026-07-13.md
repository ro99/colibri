# GLM-5.2 on 3× GPU (5060 Ti 16 + 2× 3090 24), 256 GiB RAM — tok/s investigation

This note records the July 13, 2026 investigation into why single-request decode
on our 3-GPU box sits at **~0.56 tok/s** and what it would take to reach the
**5 tok/s** target. It exists to preserve the measured facts and the dead ends
so the work can continue without repeating them.

## Machine (this box)

| Component | Spec |
|---|---|
| GPU 0 | RTX 5060 Ti 16 GiB (sm_120) |
| GPU 1 | RTX 3090 24 GiB (sm_86) |
| GPU 2 | RTX 3090 24 GiB (sm_86) |
| Total VRAM | ~64 GiB |
| RAM | 256 GiB |
| CPU | 56 logical cores (AVX2/-march=native build), 28 used |
| Model | `mateogrgic/GLM-5.2-colibri-int4-with-int8-mtp` at `c/glm52` (358 GiB, int4) |

Mixed CUDA architectures => multi-arch build is mandatory:
`-gencode arch=compute_86,code=sm_86 -gencode arch=compute_120,code=sm_120`.

## Method

Reproducible fixed-prompt decode benchmark (see `c/bench/bench_decode.sh`):

```
SNAP=c/glm52 COLI_CUDA=1 COLI_GPUS=0,1,2 CUDA_EXPERT_GB=58 CUDA_DENSE=1 \
COLI_CUDA_ATTN=1 PIN=<stats> STATS=<stats> PIN_GB=180 AUTOPIN=0 RAM_GB=200 \
DRAFT=0 TOPP=0.7 TEMP=0 OMP_NUM_THREADS=28 \
PROMPT='[gMASK]<sop><|user|>Explain the water cycle in plain English in one paragraph.<|assistant|><think></think>' \
NGEN=16 ./glm 64 4 4
```

Routing-heat (`STATS`) is seeded once and reused as the `PIN` warm set so every
run sees an identical 100%-hit expert working set (no disk variance). Each run
~3 min. The engine prints a `PROFILE:` line and per-step `tok/s`. The
`bench_decode.sh` wrapper and `bench/RESULTS.md` track every experiment.

**Critical reading note:** in the `PROFILE:` line, `expert-matmul` is `t_emm` —
the **CPU** expert matmul time only. The GPU expert time hides in `other`
(`elapsed − accounted`). The phases add up ~serially, so there is little
CPU/GPU overlap today.

## Baseline (the number to beat)

```
16 tokens in 28.77s (0.56 tok/s) | expert hit 100% | RSS 133.8 GiB
PROFILE: expert-disk 0.017s | expert-matmul 18.90s | attention 4.99s | lm_head 0.40s | other 4.48s
CUDA expert tier: 2648 resident experts (50.09 GiB) | 5906 calls served from VRAM
```

Per-token breakdown (÷16):

| cost | s/token | share |
|---|---|---|
| CPU expert matmul | 1.18 | 66% |
| attention (GPU absorb) | 0.31 | 17% |
| GPU expert matmul | 0.28 | 16% |
| lm_head | 0.03 | 2% |

Only **5906 of 8914** routed rows/token are served from VRAM; the rest are
RAM-resident experts computed on **CPU**. VRAM holds 2648 experts (~50 GiB);
per token 710 experts are routed, so only ~52% land in VRAM.

## Experiments (what was tried, what happened)

All relative to the baseline above. Roll-back = keep baseline flags.

| tag | tok/s | expert-matmul | attn | other | H2D | verdict |
|---|---|---|---|---|---|---|
| **baseline** | **0.56** | 18.90 | 4.99 | 4.48 | 162 MB | best config found |
| e1 stream_experts=1 | 0.60 | 16.72 | 5.02 | 4.37 | **44.5 GiB** | marginal; H2D explodes, cancels gain |
| e2 OMP_NUM_THREADS=56 | 0.50 | 20.95 | 5.18 | 5.30 | — | worse: RAM-bandwidth contention |
| e3 CUDA_DENSE=0 | 0.15 | 19.93 | **67.1** | 21.9 | — | disaster: CPU attention = 67s |
| e4 TC_INT4=1 TC_MIN_ROWS=1 | 0.30 | 30.43 | 5.08 | 17.5 | — | WMMA overhead + RSS blowup→disk |
| e5 COLI_GPUS=0 (1 GPU) | 0.32 | 38.72 | 4.97 | 5.47 | — | only 667 VRAM experts → CPU explodes |
| e6 TOPK=4 | 0.46 | 16.15 | 5.06 | 12.9 | — | warm-set mismatch → disk thrash |
| e7 TOPK=2 | 0.59 | 11.52 | 4.99 | 10.3 | — | expert-matmul scales ~linearly w/ #experts |

Notes per experiment:
- **e1 streaming** (`COLI_CUDA_STREAM_EXPERTS=1`): the new feature in the
  uncommitted Phase-1 diff. Moves RAM-resident experts to GPU, but re-uploads
  the full expert weights every token (44.5 GiB/token H2D). PCIe tax cancels the
  compute gain. NOT a keeper as-is.
- **e2 threads**: CPU expert matmul is RAM-bandwidth bound, not core-bound; more
  threads just contend. Keep 28.
- **e3 dense off**: GPU attention (`CUDA_DENSE=1` + `COLI_CUDA_ATTN=1`) is
  essential — CPU attention is 13× slower. Do not disable.
- **e4 tensor cores**: the `grouped_s4_wmma` path only wins when the per-device
  batch ≥ `COLI_CUDA_TC_MIN_ROWS` (default 8). At S=1 decode, per-layer-per-GPU
  batches are ~3, so TC is never selected by default and forcing it (`MIN_ROWS=1`)
  adds quantize+scheduling overhead that dominates tiny matmuls.
- **e5 single GPU**: consolidating experts on one device gives batch=8 (TC
  would engage) but kills VRAM residency (667 vs 2648 experts) → CPU explodes.
  **Residency beats batch size** on this hardware.
- **e6/e7 TOPK**: fewer experts does cut CPU matmul (~linear), but changing TOPK
  invalidates the `TOPK=8` warm set → disk thrash. And even TOPK=2 only reaches
  0.59. Not a path to 5, and hurts quality.

## Diagnosis — WITHDRAWN (invalid instrumentation)

> The numbered points below were derived from the benchmark described above and are
> **WITHDRAWN**: the timer includes prefill (only 16 output tokens), `t_emm` is
> mixed CPU+GPU (`glm.c:2105`), cache weight uploads/malloc-free happen outside the
> timer and inflate `other`, the pin fixture was self-overwritten, and the VRAM
> budget was misread. None of the per-token splits or the "1.2 tok/s ceiling" are
> supported. See `c/bench/RESULTS.md` "Methodology corrections" for the corrected
> list. A corrected benchmark + honest decode-only counters are in progress.

1. ~~CPU expert matmul is the wall (66%).~~ — t_emm is mixed CPU+GPU; cannot split.
2. ~~GPU underutilized at S=1.~~ — GPU grouped time was never cleanly isolated.
3. ~~VRAM residency hardware-capped at ~52% coverage.~~ — no honest per-token
   coverage number exists yet (cache hit counter is broken).
4. ~~6×5090 not apples-to-apples.~~ — CORRECTION: 6×5090 reached 6.84 tok/s on a
   **single request with DRAFT=0**; batching was not its mechanism, and MTP was
   slower under the current kernels. So it IS a relevant single-request target and
   its method is unknown — do not assume it was streaming/batching.

## The ceiling math — WITHDRAWN

> The per-token breakdown (1.18 s CPU + 0.31 s attn + 0.28 s GPU) and the "~1.2
> tok/s if all experts on GPU" conclusion are RETRACTED. They rely on the invalid
> prefill-inclusive timer and the invalid CPU/GPU split. No ceiling can be stated
> until decode-only + per-device counters exist.

## Recommended path forward (code, not config) — REVISED

> Pause new batching work. The highest-value immediate direction is a
> preallocated, instrumented VRAM cache/streaming arena + throughput-weighted GPU
> placement, not continuous batching by itself.

- **Honest decode-only counters + corrected benchmark first** (see `c/bench/plan.md`
   and `c/bench/RESULTS.md`). Without these, no conclusion about瓶颈 is valid.
- **Preallocated VRAM cache / streaming arena**: fixed expert slots per GPU, never
   `cudaMalloc`/`cudaFree` per miss, async overwrite + metadata-only evict, separate
   weight-byte accounting, fix the broken hit counter. Compare static residency,
   direct streaming, and cache at the SAME total VRAM budget.
- **Hardware-aware placement**: benchmark W4 expert latency + pinned H2D bandwidth
   per GPU (cards + PCIe links are heterogeneous), distribute experts and streamed
   work by measured throughput, bind transfer threads to GPU-local NUMA.
- **Batching only after the above**: record routed-rows-per-expert histogram at
   S=1,4,8,16; a useful kernel must handle ragged experts (mostly 1–4 rows) — merely
   raising S won't engage the present WMMA path (gate needs ≥8 rows/expert). Report
   aggregate and per-request latency separately.
- **Keep DRAFT=0.** Revisit MTP only after multi-row expert execution is sublinear.
   `TOPP=0.7` is a quality/perf trade-off, not free.

## Follow-up: cache implemented + re-benchmarked (corrected diagnosis)

The streaming cache + resident promotion was implemented (`CUDA_EXPERT_CACHE_GB`,
`cuda_cache_touch` in glm.c). Two corrected findings:

1. The cache ALREADY promotes experts to real resident `ColiCudaTensor` handles
   (`cuda_cache_touch` -> `qt_cuda_upload` + `cuda_eligible=1`, glm.c:1461-1470),
   so cached experts take the FAST grouped `accum` path, not `host_accum`.
   The earlier "promote to resident" hypothesis was already satisfied.
2. Re-benchmark of `CUDA_EXPERT_GB=48 CUDA_EXPERT_CACHE_GB=8
   COLI_CUDA_STREAM_EXPERTS=1` (VRAM trimmed to ~56 GiB expert + 9 GiB dense < 64 GiB,
   killing the dense-tensor CUDA errors): **0.52 tok/s**, 8328 expert calls on GPU,
   GPU "other" = 23.9 s. Trimming VRAM removed the errors but did NOT speed it up.
   So the bottleneck is the GPU itself at S=1: ~2.9 ms/call for cache-resident
   experts vs ~0.76 ms/call for the permanent-resident tier. GPU is launch/overhead
   bound on tiny per-layer batches (~3-8 rows); tensor cores idle.

Net: the cache correctly offloads CPU experts to GPU but does not reach 5 tok/s,
confirming the original ceiling analysis — single-token decode is structurally
capped near ~1.2 tok/s on this 3-GPU/64 GiB box. **5 tok/s requires batched decode
(S>1) to saturate tensor cores**, not more expert-residency work. The VRAM-resident
streaming cache is a real, working improvement (it eliminated the CPU expert wall)
but the next order-of-magnitude needs batching.
